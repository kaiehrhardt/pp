# Turso (libSQL) for durable room state, Bun.redis for cross-pod pub/sub, superseding ADR-0001

ADR-0001 chose single-process in-memory state, explicitly ruling out horizontal scaling "unless a shared store is introduced later." That store is now Turso: Room/Participant/ChatMessage state is durable there, and mutations that depend on multi-row state (reveal, host reassignment) run inside short interactive transactions rather than direct in-memory mutation. Reactions stay purely ephemeral as already documented; Duel state (challenge/accept/moves/score) is *also* treated as ephemeral, living only in the owning pod's memory and relayed cross-pod via Redis when the two participants aren't on the same pod — only a Duel's trophy result is durable. Cross-pod WebSocket delivery (full-room broadcasts and point-to-point messages like kick/reaction/duel) goes through one Redis channel per room, subscribed to only by pods with local connections for that room. This removes the single-instance ceiling ADR-0001 flagged, at the cost of network round trips per state mutation and two new external dependencies that local dev/tests also depend on (no in-memory fallback path).
